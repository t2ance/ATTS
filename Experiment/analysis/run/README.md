# `analysis/run/` — Experiment Run Registry

This directory stores **all evaluation runs** produced by `core_code/eval.py`.
This README is the single source of truth for "what is this run, and why does
it exist." **Manually maintained**. When you launch a new method or rename a
directory, update the relevant table below in the same commit.

Last full sweep: 2026-04-28 (Peijia).

---

## 1. Layout

```
analysis/run/
  <benchmark>/
    <method>/
      run_YYYYMMDD_HHMMSS/
        run_config.json        # frozen launch config (orch/explore/integrate model, cache_dir, effort, seed, ...)
        progress.json          # live status (questions_completed, accuracy, cost)
        results.jsonl          # one row per qid (final answer + grade + cost breakdown)
        rounds.jsonl           # one row per orchestrator round (tool args, tool results)
        trajectories/<qid>/    # per-question rendered trajectory.md + raw artifacts
        grading/<qid>.json     # per-question judge output
        cost_vs_accuracy.{pdf,png}  # plot, regenerated by audit.py
      precache*.log            # (rare) precache-only directory, no eval run yet
```

Conventions:
- One directory under `analysis/run/<benchmark>/` = **one method (or method
  variant)**. Multiple `run_*` subdirectories under it = repeated launches of
  the same method (typically resumed runs, or variance estimates).
- Cache files (the explore candidates) live in `analysis/cache/`, NOT here.
- `run_config.json` is authoritative for what this run actually was. If the
  directory name and the config disagree, the config wins.

## 3. Method naming legend

The directory name encodes orchestrator + variant. Read it as
`<orch_model>[_<variant>]`. Common pieces:

| Token | Meaning |
|---|---|
| `sonnet` / `opus` / `haiku` | Claude Sonnet 4.6 / Opus 4.5 / Haiku 4.5 as orchestrator |
| `gpt5.2`, `gpt5.4` | GPT-5.2 / GPT-5.4 (OpenAI) as orchestrator |
| `grpo_8b_step<N>` | Our trained Qwen3-8B GRPO checkpoint at training step N |
| `qwen3_8b_sft` | SFT-only Qwen3-8B checkpoint (before GRPO) |
| `_orch` (suffix) | Orchestrator-only run; explores read from a different model's cache (typically `cache/<bench>/haiku`). Used to ablate "orchestration cost" vs "exploration cost." |
| `_no_integrate` | Method `explore-only` (the v3 control flow): orchestrator emits StructuredOutput directly without running an integrate step on candidates. |
| `_self_refine` | Iterative self-refinement on the integrator output. |
| `_socratic_self_refine` | Socratic critic + self-refine variant (procedural floor critic + feedback loop). |
| `_budget_forcing` | Token-budget forcing on the orchestrator's reasoning trace. |
| `_skywork_rerank` | Skywork RM reranker over the cached candidates. |
| `_visualprm_rerank` | VisualPRM reranker (multimodal benches only). |
| `_standalone_integrator` | No orchestrator at all; just feed all cached candidates to an integrator and synthesize. |
| `_effort_low/medium/high` | Claude `effort` API parameter (controls thinking budget). |
| `_high` / `_low` | Same as above, abbreviated for GPT-5.2. |
| `_no_thinking` | `orchestrator_thinking_budget = -1` (thinking disabled). |
| `_run<N>` | Independent variance-estimation rerun (different seed / fresh API calls). |
| `_v2` | Config variant of the same method (typically a single knob change like `num_explores=16`). The exact delta is in `run_config.json`. |
| `_training` | Run kept *as a training-data source* (trajectories used to build SFT/GRPO datasets). DO NOT DELETE. |

## 5. Where the data lives downstream

These run directories are **not just artifacts**. They feed:

- **`analysis/audit/audit.py`** reads `run_config.json` + `results.jsonl` +
  `trajectories/<qid>/` for the 7-atom analysis. New runs are auto-picked up
  if the directory follows the naming layout in §1.
- **`scripts/plot_all_methods.py`** scans this tree to render the
  cost-vs-accuracy figures for the paper.
- **`core_code/training/sft/build_sft_hle_q101_300_thinking.py`** hardcodes
  `hle/sonnet_training/run_20260409_222818/` as its data source (see the
  bold "DO NOT DELETE" note in §4.5).
- **`core_code/training/grpo/`** indirectly depends on the SFT model trained
  from `sonnet_training/`.

## 6. Maintenance contract — when you add a new run

When you launch a new method or a new variant, do these in the same commit:

1. **Choose the directory name** following the legend in §3. If you can't fit
   the new variant into the existing legend, **add a new token to §3 first**.
   Do not invent a name that nobody else can decode.
2. **Add a row** to the appropriate per-benchmark table in §4.x. Required
   columns: method directory name, one-line config description, optional
   notes (purpose, references, status).
3. **If this run will be referenced by training code, downstream scripts, or
   the paper**, add a bullet under §5 stating the reference path. This
   prevents the run from being deleted in a future cleanup.
4. **If this run replaces / supersedes an older run**, mark the old row as
   `superseded by <new_method>` rather than deleting it from the table —
   keep the audit trail.
5. **For variance reruns** (`_run1..N`): add one row per rerun with the same
   config and label them as variance samples. **Never delete a variance
   sample without verifying its `is_correct` array is identical to the
   reference run** — config equality does NOT imply result equality (LLM
   stochasticity).

When you delete a run directory, also remove its row from §4.x in the same
commit. If you don't, the README becomes a graveyard of dangling pointers.

## 7. Open questions / known gaps

- `gpqa/multi_model` (N=8) vs `gpqa/multi_model_v2` (N=16): real experimental
  difference (candidate count), but `_v2` is a poor name. Candidate rename:
  `multi_model_explores16`. Pending user decision.
- `*_orch` suffix hides the explore-cache origin. Currently always means
  "cached **haiku** explores" by convention. If we ever introduce a non-haiku
  cache for `_orch`, the suffix needs to grow (e.g. `opus_orch_sonnetcache`).
- `gpqa/opus_orch_new` and `gpqa/multi_model_effort_high_v2` were deleted on
  2026-04-28 as duplicates without verifying `is_correct` equality. They may
  have been variance samples. If variance data for those configs is needed,
  re-run is required — see §6 rule 5.
